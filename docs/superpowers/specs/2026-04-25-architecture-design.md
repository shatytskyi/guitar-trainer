# Guitar Trainer — Architecture Design

**Status:** approved (2026-04-25)
**Owner:** Claude (sole developer of this codebase)

## Context

The project started as a single-file HTML chord trainer (`chord_trainer.html`,
~1000 lines including data, CSS, and JS). Planned future scope includes:

- Chord progression training (Am → F → C → G with timing/metronome)
- Scale practice (pentatonic, modes, fretboard visualization)
- Ear training (intervals, chords, modes by listening)
- Chord recognition via microphone (Web Audio + FFT + chromagram + matching)
- User-installable PWA, primary use case is mobile (iOS/Android)
- Trilingual UI: Russian, English, Ukrainian

Constraints:

- **No backend, no API, no auth.** Static hosting only (GitHub Pages).
- **Sole developer is an LLM agent (Claude).** The architecture must
  be optimized for LLM-driven editing: small focused files, predictable
  layout, strict typing, isolated modules.
- **User is not a web developer.** Architecture choices favour mainstream
  conventions over clever-but-niche patterns, so docs/tutorials remain
  applicable.

## Goals

1. **Extensibility:** adding a new training mode is a localized change —
   one new folder under `features/`, one entry in a registry, no
   modifications to existing modes.
2. **Isolation:** each module has clear boundaries and can be reasoned
   about (and rewritten) without loading the whole codebase into context.
3. **Type-safety as a safety net:** strict TypeScript catches data-shape
   violations and undefined access at compile time, since the LLM does
   not run the code to verify.
4. **PWA from day one:** service worker, manifest, offline cache, install
   prompt — wired in before features grow on top.
5. **Trilingual from day one:** i18n infrastructure exists before strings
   proliferate; retrofitting i18n is far more painful than starting with it.

## Non-goals

- Server-side rendering, SSR-only frameworks, edge runtimes.
- Multi-tenant features (auth, sync, sharing).
- Production-grade observability (analytics/error reporting are out of
  scope for v1).
- Hand-authoring a design system. The design system is generated by
  Claude Design from this codebase + brief, then consumed here.

## Tech foundation

| Decision        | Choice                                     | Rationale                                                                                                                  |
| --------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| Build tool      | **Vite**                                   | De-facto standard for static SPAs; first-class GitHub Pages support; HMR; minimal config.                                  |
| Language        | **TypeScript (strict)**                    | Music data has hard invariants (e.g. `frets` is exactly 6). Types prevent whole classes of regressions for an LLM author.  |
| UI              | **Vanilla DOM + small components**         | App is not SPA-complex; a framework adds learning surface and dependencies for marginal benefit.                           |
| Audio output    | **Tone.js** via thin wrapper               | Already proven; isolated behind `services/audio.ts`, swappable later.                                                      |
| Audio input     | **Web Audio API native** (when added)      | Tone.js does not solve the analyzer/chromagram problem; native API gives more control.                                     |
| PWA             | **`vite-plugin-pwa` (Workbox)**            | Industry-standard SW + manifest tooling.                                                                                   |
| Routing         | **Hash-based, custom (~30 LOC)**           | GitHub Pages has no server-side routing. A router library is overkill for a flat list of features.                         |
| i18n            | **Custom (~50 LOC)**                       | Three flat dictionaries + `t(key, params)`. Libraries (i18next, etc.) bring formatters, plurals, ICU we don't need.        |
| Tests           | **Vitest** for `shared/lib/` only          | Music-theory functions are pure and benefit most from tests; UI is rebuilt under us cheaply.                               |
| Deployment      | **GitHub Actions → GitHub Pages**          | `actions/deploy-pages` is the official path. No third-party tokens.                                                        |
| Design system   | **Claude Design** (Anthropic Labs)         | The user generates the design system externally and we consume it. We do NOT design tokens or components by hand.          |
| Styling layer   | **CSS variables (tokens) + plain CSS**     | Tokens-first means swapping the design system is editing one file. No CSS-in-JS, no runtime style engines.                 |

## Layered architecture

```
┌──────────────────────────────────────────────────────────────┐
│  Shell      bootstrap, top bar, language switch, hash router │
├──────────────────────────────────────────────────────────────┤
│  Features   chord-quiz · chord-browse · (later) sequences,   │
│             scales, ear-training, chord-recognition          │
├──────────────────────────────────────────────────────────────┤
│  Components FretboardDiagram, ChordCard, PillButton,         │
│             ToggleSwitch — pure presentational, no state     │
├──────────────────────────────────────────────────────────────┤
│  Domain     Note, Chord, Scale, Interval, Progression —      │
│             types and pure functions, no DOM, no audio       │
├──────────────────────────────────────────────────────────────┤
│  Services   audio (output) · microphone (input) · i18n ·     │
│             settings · pwa — adapters around browser APIs    │
├──────────────────────────────────────────────────────────────┤
│  Data       chords-basic, chords-extended, scales,           │
│             progressions — typed static datasets             │
└──────────────────────────────────────────────────────────────┘
```

**Hard rules:**

1. **Dependencies point downward only.** A higher layer may import from
   a lower layer; the reverse is forbidden. CI lint rules enforce this
   (eslint-plugin-import or a custom check) once the project grows.
2. **Domain is pure.** No imports from `components`, `services`, or
   `features`. Files compile in Node.js so unit tests don't need a DOM.
3. **Components are presentational.** They accept data, return DOM, and
   emit events. They don't read settings, don't call audio, don't fetch.
4. **Services wrap browser APIs.** Tone.js, AudioContext, localStorage,
   service worker, navigator.language — each behind a thin adapter so
   the rest of the code is testable and swappable.
5. **Data is static.** TypeScript modules exporting typed arrays. If we
   later move to JSON files or a generator script, only this layer changes.

## Music notation principle

**Note and chord names use international (Latin) notation everywhere in
data and domain.** No `rootRu`, no `typeName`. A chord is identified by:

- `root: 'A' | 'A#' | 'B' | 'C' | 'C#' | ... | 'G#'` — sharps only, no flats.
- `type: '' | 'm' | '7' | 'maj7' | 'm7' | 'sus2' | 'sus4' | 'add9'` —
  international suffix.

Display name is computed: `root + type` → `"A"`, `"Am"`, `"F#m7"`, `"Cmaj7"`.
Localized text exists only as optional UI decoration via i18n keys
(e.g. `chord.type.m → "minor" / "минор" / "мінор"`), derived on the fly.

This keeps domain language-neutral, aligns with Tone.js note format
(`"E2"`, `"A#3"`), and removes a class of i18n bookkeeping from data.

## The Feature contract

```ts
// src/shared/lib/feature.ts
export interface Feature {
  /** Stable id; used for hash-routing and persisted lastFeatureId. */
  readonly id: string;
  /** i18n key for the human-readable title shown in tabs. */
  readonly titleKey: string;

  /** Mount this feature into the host element with shared services. */
  mount(host: HTMLElement, ctx: FeatureContext): void;

  /** Tear down: remove listeners, stop audio, release timers, clear DOM. */
  unmount(): void;

  /** Optional: re-render in response to set/lang change while mounted. */
  onContextChange?(ctx: FeatureContext): void;
}

export interface FeatureContext {
  set: ChordSet;            // 'basic' | 'extended'
  lang: Lang;               // 'ru' | 'en' | 'uk'
  audio: AudioOutput;
  i18n: Translator;
  settings: SettingsStore;
}
```

**Lifecycle:**

1. User clicks a tab → `location.hash` changes (e.g. `#browse`).
2. Router calls `unmount()` on the active feature.
3. Router calls `mount(host, ctx)` on the new feature.
4. When `set` or `lang` changes (top bar control), router calls
   `onContextChange(ctx)` on the active feature.

**Adding a new feature:**

1. Create `src/features/<id>/index.ts` exporting a `Feature`.
2. Append it to `src/features/registry.ts`.
3. Add `feature.<id>.title` (and any other keys) to `ru.ts`, `en.ts`, `uk.ts`.

Done. Shell auto-renders the new tab.

## Foundation services

### PWA (`services/pwa.ts`)

- `vite-plugin-pwa` with Workbox under the hood.
- Precache: all bundled assets (HTML, JS, CSS, icons, fonts, JSON).
- Runtime cache: `CacheFirst` for Google Fonts and Tone.js CDN
  (so the first load stocks the cache; subsequent loads work offline).
- Manifest fields: `name`, `short_name`, `start_url`, `display: standalone`,
  `theme_color`, `background_color`, `icons` (192/512/maskable).
- Update flow: when SW detects a new version, post a non-blocking toast
  in the shell ("Update available" + button). User-driven, not forced reload.

### i18n (`services/i18n/`)

- Three flat dictionaries (`ru.ts`, `en.ts`, `uk.ts`), shape `Record<string, string>`.
- API:
  - `t(key: string, params?: Record<string, string | number>): string`
  - `setLang(lang: Lang): void`
  - `onLangChange(cb: (lang: Lang) => void): () => void` (returns unsubscribe)
- Initial language: `settings.lang` if present, else `navigator.language`,
  else `'en'`.
- Components requiring localized strings subscribe in `mount()`,
  unsubscribe in `unmount()`.
- Note/chord names are NOT localized (international notation).

### Settings (`services/settings.ts`)

- localStorage key: `guitar-trainer.settings.v1` (version embedded for
  future migration; if v2 introduces incompatible fields, write a migrator).
- Schema:
  ```ts
  interface Settings {
    lang: Lang;
    set: ChordSet;
    hideDiagram: boolean;
    theme: ThemeId;
    lastFeatureId: string;
  }
  ```
- API: `get()`, `set(partial)`, `subscribe(cb)`.
- On read: validate. If anything is malformed, fall back to defaults
  and overwrite. Never throw on bad localStorage.

### Audio output (`services/audio.ts`)

- Thin wrapper over Tone.js.
- API: `playNotes(notes: string[], options?: { strumDelay?: number }): Promise<void>`.
- Lazy `Tone.start()` on first call (browser autoplay policy requires
  a user gesture).
- Internal config (`oscillator type`, `envelope`, `volume`) lives here,
  not in features.

### Design system (`styles/tokens.css` + `shared/components/`)

The design system is **not authored in this repo**. It is generated by
[Claude Design](https://www.anthropic.com/news/claude-design-anthropic-labs)
(Anthropic Labs, launched April 2026) from this codebase plus a brief, then
delivered back as design tokens and component specifications.

**Tokenized-first principle:**
Every visual property (color, font, spacing, radius, shadow, motion) is
expressed as a CSS variable in `src/styles/tokens.css`. Components consume
variables, never hard-coded values. Replacing the design system is editing
this one file (plus possibly adding new components).

```css
/* Bad — what we never do */
.pill-button { background: #f0e6d2; padding: 4px 9px; }

/* Good — token-driven */
.pill-button {
  background: var(--surface-default);
  padding: var(--space-1) var(--space-3);
  border-radius: var(--radius-pill);
  font: var(--text-mono-xs);
}
```

**Token categories — direction, not directive.** Claude Design owns the
final shape of the token system. The list below is what the placeholder
`tokens.css` ships with in v1, so components have something to consume.
Claude Design is welcome to rename, regroup, add, drop, or restructure:

- Color (semantic, not numeric)
- Typography (families, sizes, weights, line-heights)
- Spacing scale
- Radius scale
- Shadows
- Motion (easings, durations)

**Hard constraints we ask Claude Design to respect:**

- Use **CSS variables** (or a format trivially convertible to CSS variables).
- **Semantic color names** ("surface", "accent") rather than raw scales
  ("gray-100", "red-500").
- **No CSS-in-JS / runtime style engines.**
- **Mobile-first** breakpoints; viewport min ~360 px.

Everything else — naming conventions, scale step counts, dark-theme
support, the granularity of the spacing scale, multi-tier semantic
aliasing — is up to Claude Design.

**Component inventory — functional needs, not naming directive.**
The features of v1 require UI for the responsibilities below. Claude
Design may merge, split, or rename these freely. For example, it might
unify `PillButton` and `GhostButton` under a single `Button` with a
`variant` prop, or factor common chrome into a `Surface` primitive.
We will refactor `shared/components/` to match whatever taxonomy the
design system proposes.

| Functional need                                                          | v1 placeholder name |
| ------------------------------------------------------------------------ | ------------------- |
| Full-bleed page frame with safe-area insets for iOS PWA                  | `AppShell`          |
| Title + language switch + chord-set switch                               | `TopBar`            |
| Feature tabs (quiz / browse / …)                                         | `TabBar`            |
| Decorated panel hosting feature content                                  | `Stage`             |
| Compact monospace toggle pill (3 states: default, active, highlight)     | `PillButton`        |
| Solid action button (Play, Next)                                         | `PrimaryButton`     |
| Outlined secondary action                                                | `GhostButton`       |
| Square icon-only button                                                  | `IconButton`        |
| iOS-style on/off toggle                                                  | `ToggleSwitch`      |
| Composite: chord name + meta + diagram + controls                        | `ChordCard`         |
| Grid card showing a root note and its available types                    | `RootTile`          |
| SVG renderer for chord shapes (strings, frets, finger dots, barres)      | `FretboardDiagram`  |
| Non-blocking notifications (e.g. update toast)                           | `Toast`             |
| Tap-to-reveal placeholder when diagram is hidden                         | `RevealOverlay`     |

`FretboardDiagram` is a special case: its colors come from tokens via
`fill="var(--ink)"` etc. The design system can specify what those tokens
are; the SVG-construction logic itself is domain-agnostic.

**Refactor budget at integration time.** Phase 2 (applying Claude
Design output) explicitly allows component renames, prop API changes,
and folder reorganization within `shared/components/`. Features must
adapt to the new component API; that is part of integration work, not
a regression.

**Design brief** (`docs/design-brief.md`) describes the product, audience,
and current visual direction (warm-paper / vintage-notebook). It is the
input the user hands to Claude Design alongside the source code.

**Integration milestone:** between v1 (foundations + features ported with
placeholder tokens) and v2 (any new feature), the user runs Claude Design,
returns its output, and Claude Code:

1. Replaces `src/styles/tokens.css` with the generated tokens — variable
   names may differ from v1 placeholders. All consumers are updated to
   match.
2. Adds, renames, removes, or restructures components in `shared/components/`
   to match the proposed taxonomy. Features are updated to the new
   component APIs. This is expected, not a deviation.
3. Updates `docs/design-brief.md` to record what was actually adopted vs.
   what changed during integration.

**Non-negotiable boundaries** during integration (these protect the rest
of the architecture):

- The `Feature` contract (`src/shared/lib/feature.ts`) does not change.
- The layered architecture (Domain / Components / Services / Data) is
  preserved — components stay presentational, no business logic leaks in.
- Feature internal layout (`index.ts` / `state.ts` / `view.ts`) is preserved.
- TypeScript strictness is preserved.

### Microphone (`services/microphone.ts`) — later

Out of v1 scope. Will provide:

- `start(): Promise<void>` — request mic permission, set up `AudioContext`
  + `AnalyserNode`, start FFT loop.
- `stop(): void`.
- `onChromaFrame(cb: (chroma: Float32Array) => void): () => void` —
  emit normalized 12-bin chromagram per frame.
- A separate domain function `matchChord(chroma): { chord: Chord; confidence: number }[]`
  lives in `shared/lib/` and is pure — testable without a microphone.

## Routing

Custom hash router (~30 LOC), no library:

```ts
window.addEventListener('hashchange', () => activate(parseHash()));
function parseHash(): string { return location.hash.slice(1) || features[0].id; }
function activate(id: string) {
  if (active && active.id === id) return;
  active?.unmount();
  active = features.find(f => f.id === id) ?? features[0];
  active.mount(host, ctx);
  settings.set({ lastFeatureId: active.id });
}
```

On boot: navigate to `#${settings.get().lastFeatureId}` if present,
else `#${features[0].id}`. Deep-links work because GitHub Pages serves
`index.html` for the SPA path and the hash is client-side only.

## TypeScript configuration

```jsonc
// tsconfig.json — strict end of the dial
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true
  }
}
```

`noUncheckedIndexedAccess` is the most important flag: without it,
`shape.frets[7]` is typed `number | null` instead of `number | null | undefined`,
and the LLM will write code that crashes only at runtime.

## Author-rules (LLM-developer-specific)

Encoded in CLAUDE.md as obligations, not suggestions:

1. **File size:** target ≤ 300 lines, hard ceiling 500. When a file
   approaches the ceiling, split before adding.
2. **Feature internal layout:** every `features/<id>/` contains exactly
   `index.ts`, `state.ts`, `view.ts`. No other names. If a feature needs
   more, that is a sign it should be decomposed.
3. **`shared/lib/` purity:** no imports from `components`, `services`,
   `features`. Enforced by lint rule (or manual review until linter is in).
4. **No `any`.** If a type is genuinely unknown, use `unknown` and narrow.
5. **Comments:** WHY only. Never paraphrase code. Never reference
   tickets/sessions ("added for X", "fix for Y"). Banner comments
   like the current chord_trainer.html are allowed only at file top
   for non-obvious architecture.
6. **Tests:** every public function in `shared/lib/` has at least one
   test, in a sibling `*.test.ts`. Run `npm run test` before claiming a
   change is done.
7. **CLAUDE.md is load-bearing.** Read it at the start of every session.
   Update it whenever architectural invariants change.
